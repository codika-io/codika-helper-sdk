## Failures

### CLI Bugs (real issues)

#### auth.md — [N] Login with empty API key
**Expected**: Exit non-zero, error about empty key
**Actual**: Exit 0, drops into interactive "API key:" prompt
**Classification**: CLI bug
**Root cause**: `--api-key ""` passes an empty string which bypasses the flag-present check and triggers interactive mode
**Fix**: Validate that `--api-key` value is non-empty before proceeding

#### get-skills.md — [N] Invalid API key error message
**Expected**: Error containing "unauthorized" or similar
**Actual**: Error says "Failed to fetch skills: undefined"
**Classification**: CLI bug
**Root cause**: Skills client doesn't properly extract API error response body. Affects all error paths (invalid key, wrong scope, cross-org, nonexistent instance).
**Fix**: Parse response body in skills client error handler

#### integration.md — [P] All integration set commands fail (Batch 3)
**Expected**: Integration created successfully with n8nCredentialId
**Actual**: n8n API error: `request.body.data requires property "allowedDomains"`
**Classification**: CLI/backend bug (HIGH)
**Root cause**: n8n updated its credential creation schema to require an `allowedDomains` field. The `createIntegrationPublic` Cloud Function doesn't include this field in the n8n API request body.
**Impact**: ALL integration creation is blocked — 11 tests failed. Delete tests also failed because they depend on creating an integration first.
**Fix**: Update `createIntegrationPublic` Cloud Function to include `allowedDomains` in the n8n credential creation payload.

#### init.md — [P] --with-data-ingestion generates invalid config.ts (Batch 4)
**Expected**: Valid TypeScript in scaffolded config.ts
**Actual**: Escaped backticks (`\``) in the data ingestion webhooks section produce invalid TypeScript
**Classification**: CLI bug
**Root cause**: Template generator escapes backticks that should be raw for template literal strings in the DI config section
**Fix**: Use raw backticks in the DI config template in `src/cli/templates/config-template.ts`

#### deploy-data-ingestion.md — Scope enforcement gap (Batch 4)
**Expected**: Key with only `deploy:use-case` (no `deploy:data-ingestion`) should be rejected
**Actual**: Request passed scope check, reached n8n, failed at n8n level (not scope enforcement)
**Classification**: Possible Cloud Function bug
**Root cause**: `deployProcessDataIngestionPublic` may not check for `deploy:data-ingestion` scope separately, or `deploy:use-case` implicitly covers DI deployment
**Fix**: Investigate Cloud Function scope enforcement for DI endpoint

#### organization-create-key.md — [S] SECURITY: No scope escalation prevention (Batch 5)
**Expected**: A key with `api-keys:manage` + `projects:read` should NOT be able to create a key with `deploy:use-case` scope (scope the caller doesn't have)
**Actual**: Key created successfully with `deploy:use-case` scope despite caller lacking it
**Classification**: **SECURITY BUG (HIGH)**
**Root cause**: `createOrganizationApiKeyPublic` Cloud Function does not verify that requested scopes are a subset of the caller's scopes. Any key with `api-keys:manage` can grant any scope.
**Impact**: Privilege escalation — a restricted key holder can create a fully-privileged key
**Fix**: Add scope subset validation in the Cloud Function before creating the key

#### organization-create.md — Logo upload fails (Batch 5, infrastructure)
**Expected**: Logo uploaded to GCS and URL returned
**Actual**: GCS error: "Cannot update access control for an object when uniform bucket-level access is enabled"
**Classification**: Infrastructure issue (not CLI bug)
**Root cause**: GCS bucket has uniform bucket-level access enabled, but the Cloud Function tries to set per-object ACLs
**Fix**: Update Cloud Function to use uniform access patterns or switch to signed URLs
